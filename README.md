# GenesisAI: Разработка, управляемая ИИ-агентами

## Введение

**GenesisAI** — это стартовый репозиторий, предназначенный для быстрой инициализации проектов с помощью интеллектуальных агентов ИИ. Его основная цель — автоматизировать процесс создания всесторонней документации, спецификаций и, в будущем, самого кода, позволяя разработчикам сосредоточиться на инновациях и уникальных аспектах своего проекта.

## Начало работы и запуск агентов

Для запуска агентов, которые генерируют документацию и спецификации для вашего проекта, выполните следующую команду:

```bash
npm run generate-docs
```

Эта команда активирует оркестратор агентов, который проанализирует ваши высокоуровневые требования и создаст или обновит соответствующие файлы документации в папке `docs/` (например, спецификации для фронтенда, бэкенда, user-flow и другие).

После генерации документации вы сможете её просмотреть и при необходимости внести корректировки, давая обратную связь агентам.

## Конфигурация и управление агентами

Для эффективного взаимодействия с агентами и управления процессом генерации, важно знать следующие ключевые файлы:

*   **Описание задачи**: Основная задача, которую должен выполнить агент (например, что нужно задокументировать или какая спецификация требуется), описывается человеческим языком в файле:
[current-task-description.md](agents/doc_agents/config/current-task-description.md)
Именно этот файл является отправной точкой для агентов, понимающих ваши требования.
**ВАЖНО!**: тут первой строкой пишем путь и название папки, например docs/jira/auth

*   **Файлы контекста и правила**: Если для выполнения задачи агентам необходимы дополнительные файлы для контекста (например, уже существующий код, примеры, или дополнительные правила), а также специфические правила выполнения, их можно указать в конфигурационном файле:
[task-config.json](agents/doc_agents/config/task-config.json)
    *   **Описание полей `task-config.json`**:
        *   `topic`: Определяет общую тему или заголовок для задачи, которая будет документироваться. Используется для категоризации и представления документации.
        *   `outputTaskDocsDir`: Указывает директорию, куда будут сохраняться сгенерированные документы по задаче. Если оставить пустым, будут использоваться настройки по умолчанию или текущая рабочая директория.
        *   `taskDescriptionFile`: Путь к файлу, содержащему описание текущей задачи. Этот файл используется агентом для понимания сути задачи и генерации соответствующей документации.
        *   `allowAdditionalFiles`: Булево значение, указывающее, разрешено ли агенту включать дополнительные файлы в контекст задачи, если это необходимо для более полного понимания или генерации документации.
        *   `maxIterations`: Максимальное количество итераций, которое агент может выполнить при обработке задачи. Это предотвращает бесконечные циклы и ограничивает время выполнения.
        *   `dialogueLogFile`: Путь к файлу, в котором будет вестись журнал диалогов агента. Используется для отслеживания хода выполнения задачи и взаимодействия с другими агентами или пользователем.
        *   `primaryAgentPromptFile`: Путь к файлу, содержащему основной промпт (инструкции) для первичного агента. Определяет его поведение и задачи.
        *   `secondaryAgentPromptFile`: Путь к файлу, содержащему промпт для вторичного агента. Определяет его вспомогательные функции и задачи.
        *   `docsReadmeFile`: Имя файла README, который будет использоваться для документации. Обычно это основной файл, описывающий проект или его часть.
        *   `additionalContextFiles`: Массив строк, содержащих пути к дополнительным файлам, которые агент должен учитывать в качестве контекста при обработке задачи. Может включать файлы с кодом, конфигурацией и т.д.
        *   `additionalContextDirectories`: Массив строк, содержащих пути к дополнительным директориям, которые агент должен сканировать для получения контекста. Полезно для автоматического сбора информации из связанных модулей или компонентов.
        *   `primaryAgentRuleFiles`: Массив строк, содержащих пути к файлам с правилами для первичного агента. Эти правила определяют его логику, ограничения и приоритеты.
        *   `secondaryAgentRuleFiles`: Массив строк, содержащих пути к файлам с правилами для вторичного агента. Аналогично, эти правила управляют поведением вторичного агента.
        *   `maxContextChars`: Максимальное количество символов, которое может быть использовано для общего контекста задачи. Помогает контролировать размер входных данных для моделей ИИ и избегать переполнения контекста.
        *   `maxAdditionalContextChars`: Максимальное количество символов для дополнительного контекста (из `additionalContextFiles` и `additionalContextDirectories`). Используется для более тонкого контроля над объемом передаваемой информации.
        *   `tokenReportInterval`: Интервал, с которым будет генерироваться отчет по использованию токенов. Например, `'before_review'` означает, что отчет будет создан перед этапом ревью.
        *   `maxTokensPerAgentCall`: Максимальное количество токенов, которое может быть использовано за один вызов агента. Ограничивает потребление ресурсов и стоимость.
        *   `writerModel`: Идентификатор модели ИИ, используемой для написания документации. Например, `'gpt-4.1'` указывает на конкретную версию модели GPT.
        *   `reviewerModel`: Идентификатор модели ИИ, используемой для ревью (проверки) сгенерированной документации. Обычно это та же или похожая модель, что и `writerModel`, для обеспечения согласованности и качества.

*   **Основной скрипт оркестратора**: Запуск всего процесса генерации документации или спецификаций осуществляется через главный скрипт оркестратора:
[doc-agent-orchestrator.j](agents/doc_agents/orchestrator/doc-agent-orchestrator.js)

*   **Правила и промты для агентов**: Сами инструкции, правила поведения и промты (запросы), которыми руководствуются агенты при выполнении своих задач, определены в папке:
[prompts](agents/doc_agents/prompts/)
    Здесь вы найдете логику, которая формирует их "мышление" и подходы к работе.

## Будущее GenesisAI

В ближайших планах `GenesisAI` — расширение функциональности агентов для автоматической генерации рабочего кода на основе утвержденных спецификаций. Это позволит создать полностью автоматизированный цикл разработки от идеи до готового продукта, минимизируя ручное вмешательство и ускоряя процесс.
